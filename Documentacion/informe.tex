\section{Introducci\'on}

El presente documento describe el dise\~no e implementaci\'on de un \textbf{prototipo acad\'emico de videoconferencia} inspirado en Zoom, desarrollado en Python como parte del curso de Programaci\'on Orientada a Objetos (PC4).

El sistema emplea una arquitectura \textbf{cliente-servidor} basada en \textbf{sockets TCP} con un protocolo de mensajes en formato \textbf{JSON}. Cuenta con autenticaci\'on de usuarios, creaci\'on y gesti\'on de salas virtuales, chat en tiempo real, transferencia de archivos y transmisi\'on de video desde la c\'amara web.

\section{Objetivos}

\begin{itemize}
    \item Aplicar los principios de la POO: herencia, encapsulamiento, polimorfismo.
    \item Implementar una arquitectura cliente-servidor mediante sockets TCP.
    \item Desarrollar un protocolo de comunicaci\'on propio basado en JSON.
    \item Integrar m\'ultiples funcionalidades: login, salas, chat, archivos y video.
    \item Utilizar SQLite para la persistencia de informaci\'on.
    \item Emplear concurrencia con \texttt{threading} para atender m\'ultiples clientes.
\end{itemize}

\section{Tecnolog\'ias Utilizadas}

\begin{itemize}
    \item \textbf{Lenguaje:} Python 3.10+
    \item \textbf{Interfaz gr\'afica:} Tkinter
    \item \textbf{Comunicaci\'on:} Sockets TCP
    \item \textbf{Protocolo:} JSON sobre TCP
    \item \textbf{Base de datos:} SQLite
    \item \textbf{Concurrencia:} \texttt{threading}
    \item \textbf{Video:} OpenCV o ffmpeg
    \item \textbf{Im\'agenes:} Pillow

\subsubsection{ManejadorCliente}
\texttt{Servidor/manejador\_cliente.py}

Maneja la comunicaci\'on con un cliente en un hilo separado. El bucle principal lee mensajes JSON delimitados por \verb|\n| y los despacha seg\'un su tipo:

\begin{lstlisting}
class ManejadorCliente:
    def iniciar(self):
        buffer = ""
        while self._conectado:
            datos = self._conexion.recv(4096).decode("utf-8")
            buffer += datos
            while "\n" in buffer:
                linea, buffer = buffer.split("\n", 1)
                mensaje = json.loads(linea)
                self._procesar_mensaje(mensaje)

    def _procesar_mensaje(self, mensaje):
        tipo = mensaje.get("type")
        if tipo == Protocolo.LOGIN_REQUEST:
            self._manejar_login(mensaje)
        elif tipo == Protocolo.CREATE_ROOM:
            self._manejar_crear_sala(mensaje)
        elif tipo == Protocolo.CHAT_MESSAGE:
            self._servidor.reenviar_a_sala(
                mensaje.get("roomCode"), mensaje, self)
\end{lstlisting}

\subsubsection{Servidor}
\texttt{Servidor/servidor.py}

Servidor TCP principal. Acepta conexiones y crea un hilo por cada cliente:

\begin{lstlisting}
class Servidor:
    def iniciar(self):
        BaseDatos().inicializar()
        self._socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        self._socket.bind((self._host, self._puerto))
        self._socket.listen()
        while self._activo:
            conexion, direccion = self._socket.accept()
            manejador = ManejadorCliente(self, conexion, direccion)
            hilo = threading.Thread(target=manejador.iniciar, daemon=True)
            hilo.start()
\end{lstlisting}

class ClienteSocket:
    def _escuchar(self):
        buffer = ""
        while self._conectado:
            datos = self._socket.recv(4096).decode("utf-8")
            buffer += datos
            while "\n" in buffer:
                linea, buffer = buffer.split("\n", 1)
                mensaje = json.loads(linea)
                tipo = mensaje.get("type")
                cb = self._callbacks.get(tipo)
                if cb:
                    cb(mensaje)

    def enviar(self, datos):
        self._socket.send(
            (json.dumps(datos) + "\n").encode("utf-8"))
\end{lstlisting}

\section{Pilares de la Programaci\'on Orientada a Objetos}

El sistema aplica los cuatro pilares fundamentales de la POO, detallados a continuaci\'on con fragmentos de c\'odigo extra\'idos del proyecto.

\subsection{Encapsulamiento}

Todas las clases emplean atributos privados (prefijo \texttt{\_}) y exponen \'unicamente lo necesario mediante propiedades de solo lectura o m\'etodos p\'ublicos.

\noindent\textbf{Ejemplo:} \texttt{Cliente/modelos/usuario.py}

\begin{lstlisting}
class Usuario:
    def __init__(self, id_usuario, nombres, correo, rol):
        self._id_usuario = id_usuario
        self._nombres = nombres
        self._correo = correo
        self._rol = rol

    @property
    def id_usuario(self):
        return self._id_usuario

    @property
    def nombres(self):
        return self._nombres

    def es_host(self):
        return self._rol.lower() == "host"
\end{lstlisting}

Los atributos \texttt{\_id\_usuario}, \texttt{\_nombres}, \texttt{\_correo} y \texttt{\_rol} solo son accesibles mediante \texttt{@property}. La l\'ogica de verificaci\'on de rol queda encapsulada en \texttt{es\_host()}.

\noindent\textbf{Ejemplo:} \texttt{Servidor/servidor.py}

\begin{lstlisting}
class Servidor:
    def __init__(self, host="0.0.0.0", puerto=5000):
        self._host = host
        self._puerto = puerto
        self._socket = None
        self._activo = False
        self._clientes = []
        self._salas = {}
        self._lock = threading.Lock()

    def iniciar(self):
        ...
    def detener(self):
        ...
    def reenviar_a_sala(self, codigo_sala, mensaje, emisor=None):
        ...
\end{lstlisting}

El estado interno del servidor es privado; la interacci\'on externa ocurre solo a trav\'es de los m\'etodos p\'ublicos \texttt{iniciar()}, \texttt{detener()}, \texttt{remover\_cliente()} y \texttt{reenviar\_a\_sala()}.

\subsection{Polimorfismo}

El polimorfismo se manifiesta en tres \'areas clave del sistema.

\noindent\textbf{Frames intercambiables:} En \texttt{MainApp}, la variable \texttt{self.\_frame\_actual} puede contener cualquier subclase de \texttt{tk.Frame} (\texttt{LoginFrame}, \texttt{PantallaPrincipal}, \texttt{PantallaSala}) y se invocan m\'etodos comunes como \texttt{pack()} y \texttt{destroy()} sin conocer el tipo concreto.

\begin{lstlisting}
class MainApp(tk.Tk):
    def _mostrar_login(self):
        if self._frame_actual:
            self._frame_actual.destroy()
        self._frame_actual = LoginFrame(self, ...)
        self._frame_actual.pack(fill=tk.BOTH, expand=True)

    def _mostrar_principal(self):
        if self._frame_actual:
            self._frame_actual.destroy()
        self._frame_actual = PantallaPrincipal(self, ...)
        self._frame_actual.pack(fill=tk.BOTH, expand=True)
\end{lstlisting}

\noindent\textbf{Backends de c\'amara:} Los m\'etodos \texttt{\_capturar\_con\_opencv()} y \texttt{\_capturar\_con\_ffmpeg()} implementan la misma interfaz conceptual y se seleccionan en tiempo de ejecuci\'on.

\begin{lstlisting}
def _capturar_y_enviar(self):
    if self._backend_video == "opencv":
        self._capturar_con_opencv()
    else:
        self._capturar_con_ffmpeg()
\end{lstlisting}

\noindent\textbf{Despacho de mensajes:} El servidor procesa todos los mensajes entrantes con un \'unico m\'etodo que deriva cada tipo a su manejador espec\'ifico.

\begin{lstlisting}
def _procesar_mensaje(self, mensaje):
    tipo = mensaje.get("type")
    if tipo == Protocolo.LOGIN_REQUEST:
        self._manejar_login(mensaje)
    elif tipo == Protocolo.CREATE_ROOM:
        self._manejar_crear_sala(mensaje)
    elif tipo == Protocolo.CHAT_MESSAGE:
        self._manejar_chat(mensaje)
    elif tipo == Protocolo.LEAVE_ROOM:
        self._sala_actual = None
\end{lstlisting}

\section{Patrones de Dise\~no Implementados}

A continuaci\'on se describen los patrones de dise\~no identificados en la implementaci\'on del sistema.

\subsection{Singleton}

\emph{Propósito}: Garantizar que una clase tenga una \'unica instancia y proporcionar un punto de acceso global a ella.

\medskip
\noindent\textbf{Implementaci\'on}: \texttt{Servidor/base\_datos.py}

La clase \texttt{BaseDatos} sobreescribe \texttt{\_\_new\_\_} para retornar siempre la misma instancia. La inicializaci\'on del esquema se ejecuta una sola vez mediante una bandera \texttt{\_inicializada}.

\begin{lstlisting}
class BaseDatos:
    _instancia = None

    def __new__(cls, *args, **kwargs):
        if cls._instancia is None:
            cls._instancia = super().__new__(cls)
        return cls._instancia
\end{lstlisting}

Todas las llamadas a \texttt{BaseDatos()} desde \texttt{manejador\_cliente.py} y \texttt{auth\_service.py} obtienen la misma instancia \'unica.

\subsection{Observer}

\emph{Propósito}: Definir una dependencia de uno a muchos entre objetos, de modo que cuando un objeto cambie su estado, todos sus dependientes sean notificados autom\'aticamente.

\medskip
\noindent\textbf{Implementaci\'on}: \texttt{Cliente/cliente\_socket.py}

La clase \texttt{ClienteSocket} mantiene un diccionario \texttt{\_callbacks} que asocia tipos de mensaje con funciones manejadoras. Cuando llega un mensaje desde el servidor, se invoca la funci\'on registrada para ese tipo:

\begin{lstlisting}
class ClienteSocket:
    def registrar_callback(self, tipo, callback):
        self._callbacks[tipo] = callback

    def _escuchar(self):
        while self._conectado:
            mensaje = json.loads(linea)
            tipo = mensaje.get("type")
            cb = self._callbacks.get(tipo)
            if cb:
                cb(mensaje)
\end{lstlisting}

Las pantallas (\texttt{PantallaPrincipal}, \texttt{PantallaSala}) se suscriben como observadores mediante \texttt{registrar\_callback()} y se desuscriben al salir con \texttt{remover\_callback()}.

\subsection{Strategy}

\emph{Propósito}: Definir una familia de algoritmos, encapsular cada uno y hacerlos intercambiables.

\medskip
\noindent\textbf{Implementaci\'on}: \texttt{Cliente/pantallas/pantalla\_sala.py}

El sistema selecciona en tiempo de ejecuci\'on entre dos estrategias de captura de video: \texttt{\_capturar\_con\_opencv()} y \texttt{\_capturar\_con\_ffmpeg()}. El m\'etodo \texttt{\_detectar\_backend()} elige la estrategia disponible y \texttt{\_capturar\_y\_enviar()} delega en ella:

\begin{lstlisting}
def _detectar_backend(self):
    try:
        import cv2
        self._backend_video = "opencv"
    except ImportError:
        self._backend_video = "ffmpeg"

def _capturar_y_enviar(self):
    if self._backend_video == "opencv":
        self._capturar_con_opencv()
    else:
        self._capturar_con_ffmpeg()
\end{lstlisting}

\subsection{Facade}

\emph{Propósito}: Proporcionar una interfaz unificada para un conjunto de interfaces de un subsistema.

\medskip
\noindent\textbf{Implementaci\'on}: \texttt{Servidor/manejador\_cliente.py}

\texttt{ManejadorCliente} act\'ua como una fachada que orquesta tres subsistemas: \texttt{AuthService} (autenticaci\'on), \texttt{BaseDatos} (persistencia) y \texttt{Servidor} (difusi\'on en salas). El exterior (\texttt{servidor.py}) solo invoca \texttt{manejador.iniciar()}, y la fachada se encarga de despachar cada mensaje al subsistema correspondiente.

\subsection{Command}

\emph{Propósito}: Encapsular una solicitud como un objeto, permitiendo parametrizar clientes con distintas solicitudes.

\medskip
\noindent\textbf{Implementaci\'on}: \texttt{Servidor/protocolo.py} y \texttt{Servidor/manejador\_cliente.py}

El protocolo define constantes (\texttt{LOGIN\_REQUEST}, \texttt{CREATE\_ROOM}, \texttt{JOIN\_ROOM\_REQUEST}, etc.) que act\'uan como comandos. \texttt{ManejadorCliente.\_procesar\_mensaje()} funciona como despachador: lee el campo \texttt{type} del mensaje JSON y lo enruta al manejador correspondiente.

\subsection{Template Method}

\emph{Propósito}: Definir el esqueleto de un algoritmo en una operaci\'on, delegando algunos pasos a las subclases.

\medskip
\noindent\textbf{Implementaci\'on}: \texttt{Cliente/pantallas/pantalla\_sala.py}

El ciclo de vida de la c\'amara sigue un algoritmo fijo: iniciar captura, capturar y enviar fotogramas, detener captura. Los pasos concretos (\texttt{\_capturar\_con\_opencv()} o \texttt{\_capturar\_con\_ffmpeg()}) se inyectan en ese esqueleto.

\subsection{Factory Method}

\emph{Propósito}: Definir una interfaz para crear un objeto, pero dejar que las subclases decidan qu\'e clase instanciar.

\medskip
\noindent\textbf{Implementaci\'on}: \texttt{Cliente/main.py}

\texttt{MainApp} implementa m\'etodos factor\'ia (\texttt{\_mostrar\_login()}, \texttt{\_mostrar\_principal()}, \texttt{\_mostrar\_sala()}) que destruyen el frame actual e instancian la subclase de \texttt{tk.Frame} correspondiente:

\begin{lstlisting}
def _mostrar_login(self):
    if self._frame_actual:
        self._frame_actual.destroy()
    self._frame_actual = LoginFrame(self, self._conectar)
    self._frame_actual.pack(fill=tk.BOTH, expand=True)
\end{lstlisting}

\section{Conclusiones}

\begin{itemize}
    \item Se implement\'o un prototipo funcional de videoconferencia con arquitectura cliente-servidor.
    \item Los sockets TCP con protocolo JSON proporcionan una base s\'olida para comunicaci\'on en tiempo real.
    \item Los cuatro pilares de la POO -- encapsulamiento, herencia, polimorfismo y abstracci\'on -- fueron aplicados consistentemente en todo el c\'odigo.
    \item Se identificaron y aplicaron siete patrones de dise\~no: Singleton, Observer, Strategy, Facade, Command, Template Method y Factory Method.
    \item La POO permiti\'o organizar el c\'odigo en clases especializadas con responsabilidades claras.
    \item La concurrencia con hilos permite atender m\'ultiples clientes simult\'aneamente.
    \item La transmisi\'on de video se logr\'o mediante captura, compresi\'on JPEG y codificaci\'on base64.
\end{itemize}
